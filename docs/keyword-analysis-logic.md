# 네이버 스마트스토어 상품 키워드 성과 분석 및 개선 자동화 — 로직 기획문서

> **문서 목적**: 키워드 검색 인식 전략의 핵심 규칙을 자동화 프로그램 로직에 직접 활용 가능한 형태로 정의한다.

---

## 1. 키워드 분류 체계 (Keyword Taxonomy)

프로그램이 모든 키워드를 아래 5가지 유형 중 하나로 판별할 수 있어야 한다.

| 대분류 | 유형 | 정의 | 판별 기준 |
|--------|------|------|-----------|
| **배열성** | 조합형 (Composite) | 여러 키워드가 조합되어 만들어진 키워드 | ① 붙여쓰기 등록상품수 = 띄어쓰기 등록상품수 **AND** ② 순서 변경 띄어쓰기 등록상품수도 동일 |
| **배열성** | 일체형 (Integral) | 더 이상 분리 불가능한 단일 덩어리 키워드 | ① 붙여쓰기 등록상품수 ≠ 띄어쓰기 등록상품수 |
| **배열성** | 순서고정 (Order-Fixed) | 구성 키워드의 순서가 고정된 키워드 | ① 붙여쓰기 등록상품수 = 띄어쓰기 등록상품수 **AND** ② 순서 변경 띄어쓰기 등록상품수 ≠ 원래 등록상품수 |
| **기타** | 동의어 (Synonym) | 네이버가 동일 검색으로 인식하는 서로 다른 표기 | 키워드 A 등록상품수 = 키워드 B 등록상품수 (완전 일치) |
| **기타** | 불필요 (Redundant) | 추가해도 검색 결과에 영향 없는 수식어 | "수식어 + 핵심어" 등록상품수 = "핵심어" 단독 등록상품수 |

## 2. 키워드 유형 자동 판별 로직 (Classification Algorithm)

```
입력: keyword (판별 대상 키워드)

STEP 1 ─ 분리 가능 여부 판별
  count_joined   = 등록상품수(keyword 붙여쓰기)
  count_spaced   = 등록상품수(keyword 띄어쓰기)

  IF count_joined ≠ count_spaced:
      → 일체형(Integral) 확정
      → STOP

STEP 2 ─ 순서 변경 테스트 (count_joined == count_spaced인 경우)
  count_reversed = 등록상품수(keyword 순서 반전 + 띄어쓰기)

  IF count_spaced ≠ count_reversed:
      → 순서고정(Order-Fixed) 확정
      → STOP

  IF count_spaced == count_reversed:
      → 조합형(Composite) 확정
      → STOP
```

### 보조 판별 로직

```
동의어 판별:
  IF 등록상품수(keyword_A) == 등록상품수(keyword_B) AND A ≠ B:
      → 동의어(Synonym) 관계 플래그

불필요 키워드 판별:
  IF 등록상품수("수식어 + 핵심어") == 등록상품수("핵심어"):
      → 수식어를 불필요(Redundant) 플래그
```

## 3. 유형별 상품명 작성 규칙 (Naming Rules Engine)

자동화 프로그램이 상품명 생성/검증 시 적용해야 할 규칙이다.

### 3-1. 조합형 키워드

| 규칙 ID | 규칙 | 예시 |
|---------|------|------|
| C-01 | 공통 구성요소는 **1회만** 기재하고 나머지 요소를 나열하여 조합 노출 생성 | "돼지 오리 양 염소 고기" → 돼지고기, 오리고기, 양고기, 염소고기 모두 노출 |
| C-02 | 키워드 간 **순서 무관** — 자유롭게 배치 가능 | "고기 양" = "양 고기" = "양고기" |
| C-03 | 조합형 키워드 사이에 다른 키워드가 삽입되어도 검색 인식됨 | "양 한우 고기" → "양고기" 노출 가능 |

### 3-2. 일체형 키워드

| 규칙 ID | 규칙 | 예시 |
|---------|------|------|
| I-01 | 반드시 **통째로(붙여서)** 상품명에 기재 | "소고기" → ✅ / "소 고기" 조합 기대 → ❌ |
| I-02 | 일체형 키워드의 구성 글자로 다른 키워드와 조합 노출 불가 | "꽃등심"에서 "등심"을 분리하여 "등심 구이" 노출 불가 |
| I-03 | 해당 키워드의 부분 노출이 필요하면 부분 키워드를 **별도 추가** 기재 | "꽃등심" + "등심" 별도 기재 → "꽃등심", "등심" 모두 노출 |

### 3-3. 순서고정 키워드

| 규칙 ID | 규칙 | 예시 |
|---------|------|------|
| O-01 | 구성 키워드가 **원래 순서 그대로 나란히** 배치되어야 노출 | "에어 조던" → ✅ / "조던 에어" → ❌ |
| O-02 | 구성 키워드 사이에 **다른 키워드 삽입 불가** | "에어 맥스 조던" → "에어조던" 노출 ❌ |
| O-03 | 띄어쓰기는 허용 (붙여쓰기 = 띄어쓰기 동일 인식) | "에어조던" = "에어 조던" |
| O-04 | 대부분의 **브랜드명, 제조사명**이 이 유형에 해당 | 등록 시 반드시 원래 순서 유지 |

### 3-4. 동의어 키워드

| 규칙 ID | 규칙 | 예시 |
|---------|------|------|
| S-01 | 동의어 쌍 중 **하나만 사용**하여 공간 절약 | "쓰레기통" = "휴지통" → 하나만 기재 |
| S-02 | 브랜드 동의어(한/영)는 **자동완성 레이어에서 영문 클릭 등록** 필수 | 한글 "아이르"만 등록 시 브랜드 필터링 미작동 |
| S-03 | 심화 전략: 의도적으로 동의어를 추가 사용하여 특정 필터/노출 확보 가능 | 상황에 따라 판단 |

### 3-5. 불필요한 키워드

| 규칙 ID | 규칙 | 예시 |
|---------|------|------|
| R-01 | 상품명에서 **제거**하여 공간 확보 | "예쁜", "추천", "저렴한", "가격", "판매" 등 |
| R-02 | 네이버가 무시하는 수식어이므로 사용 시 키워드 슬롯 낭비 | "예쁜 화분" = "화분" (동일 결과) |

## 4. 상품명 최적화 점수 산출 로직 (Scoring Model)

현재 상품명의 키워드 효율을 수치로 평가하는 모델이다.

### 4-1. 감점 요인 (Penalty)

| 감점 항목 | 조건 | 감점 |
|-----------|------|------|
| 불필요 키워드 사용 | R-01 위반 키워드 발견 | -5점 / 건 |
| 동의어 중복 사용 | 동의어 쌍이 모두 포함됨 | -3점 / 건 |
| 일체형 키워드 분리 기재 | I-01 위반 (띄어쓰기로 분리) | -10점 / 건 |
| 순서고정 키워드 순서 오류 | O-01 위반 (역순 배치) | -10점 / 건 |
| 순서고정 키워드 사이 삽입 | O-02 위반 (중간에 타 키워드) | -10점 / 건 |
| 조합형 키워드 불필요 반복 | C-01 미활용 (공통 요소 중복 기재) | -3점 / 건 |

### 4-2. 가점 요인 (Bonus)

| 가점 항목 | 조건 | 가점 |
|-----------|------|------|
| 조합형 키워드 공간 절약 활용 | 공통 요소 1회 기재로 N개 키워드 노출 | +3점 × (N-1) |
| 유효 키워드 밀도 | (유효 키워드 수 / 전체 키워드 수) ≥ 0.9 | +10점 |
| 일체형 키워드 정확 기재 | 일체형 키워드 통째 기재 확인 | +2점 / 건 |

### 4-3. 최종 점수

```
최적화 점수 = 기본 100점 + Σ(가점) + Σ(감점)
등급: S(95+), A(85-94), B(70-84), C(60-69), D(59이하)
```

## 5. 자동 개선 제안 로직 (Recommendation Engine)

점수 산출 후 구체적 개선 액션을 자동 생성한다.

```
FOR EACH 감점_항목 IN 상품명_분석결과:

  CASE 불필요_키워드:
    → "'{keyword}' 제거 권장. 제거 시 {N}자 공간 확보 가능."

  CASE 동의어_중복:
    → "'{keyword_A}'와 '{keyword_B}'는 동의어. '{keyword_B}' 제거 권장."

  CASE 일체형_분리:
    → "'{keyword}'는 일체형 키워드. 반드시 붙여서 기재. 현재 분리 기재로 노출 불가."

  CASE 순서고정_오류:
    → "'{keyword}'는 순서고정 키워드. '{correct_order}' 순서로 수정 필요."

  CASE 순서고정_삽입:
    → "'{keyword}' 사이에 '{inserted}'가 삽입됨. 제거하여 나란히 배치 필요."

  CASE 조합형_반복:
    → "'{common_part}' 중복 기재됨. 1회만 기재하고 조합 노출 활용 권장. {N}자 절약 가능."
```

## 6. 유효 노출량 시뮬레이션 로직

상품명 변경 전후의 예상 노출 키워드 수를 비교한다.

### 6-1. 노출 가능 키워드 추출

```
입력: product_name (상품명), keyword_db (키워드 유형 DB)

1. 상품명을 토큰(개별 키워드 단위)으로 분리
2. 가능한 모든 2~N 토큰 조합 생성
3. 각 조합에 대해:
   - keyword_db에서 유형 조회
   - 조합형: 순서 무관 조합 → 노출 가능 키워드 목록에 추가
   - 일체형: 붙여쓰기 상태로 상품명에 존재하는 경우만 추가
   - 순서고정: 원래 순서로 나란히 존재하는 경우만 추가
4. 불필요 키워드 제외, 동의어 통합
5. 최종 유효 노출 키워드 목록 반환
```

### 6-2. 변경 전후 비교

```
before_keywords = 노출가능키워드추출(현재_상품명)
after_keywords  = 노출가능키워드추출(개선_상품명)

Δ_노출 = |after_keywords| - |before_keywords|
Δ_검색량 = Σ(after 월간검색량) - Σ(before 월간검색량)

리포트 출력:
  - 추가된 노출 키워드 목록 + 각 월간 검색량
  - 제거된 노출 키워드 목록 (의도적/비의도적)
  - 총 유효 노출량 변화율(%)
```

## 7. 스토어명 연동 분석 로직

스토어명도 검색 인식 파트이므로 상품명과 결합한 추가 노출 분석이 필요하다.

```
입력: store_name, product_name

1. store_name 토큰 분리
2. store_name 키워드 유형 판별 (조합형이면 분리 가능)
3. store_name 토큰 + product_name 토큰 결합
4. 결합된 토큰으로 추가 노출 가능 키워드 탐색
5. 추가 노출 키워드 = (결합 노출) - (product_name 단독 노출)
6. "스토어명에서 파생되는 보너스 키워드" 리포트 출력
```

> **사례**: 스토어명 "늘푸름 모닝" → "늘푸른" + "모닝" 조합형 → 상품명에 "마스크" 포함 시 "늘푸른 마스크", "모닝 마스크" 추가 노출 확보

## 8. 데이터 수집 요구사항

| 데이터 | 출처 | 용도 | 수집 주기 |
|--------|------|------|-----------|
| 등록상품수 (붙여쓰기) | 네이버 쇼핑 검색 | 키워드 유형 판별 | 키워드 등록/갱신 시 |
| 등록상품수 (띄어쓰기) | 네이버 쇼핑 검색 | 키워드 유형 판별 | 키워드 등록/갱신 시 |
| 등록상품수 (순서 반전 띄어쓰기) | 네이버 쇼핑 검색 | 조합형 vs 순서고정 판별 | 키워드 등록/갱신 시 |
| 월간 검색량 | 네이버 키워드 도구 | 유효 노출량 산출 | 월 1회 |
| 동의어 매핑 테이블 | 수동 구축 + 크롤링 | 동의어 판별 | 수시 갱신 |
| 불필요 키워드 사전 | 수동 구축 (예쁜, 추천, 저렴한 등) | 불필요 키워드 필터링 | 수시 갱신 |

## 9. 프로세스 흐름 요약

```
[상품명 입력]
     │
     ▼
[토큰 분리 & 키워드 추출]
     │
     ▼
[키워드 유형 자동 판별] ──→ [키워드 DB 저장/갱신]
     │
     ▼
[유형별 작성 규칙 검증]
     │
     ▼
[최적화 점수 산출]
     │
     ├─→ [감점 항목별 개선 제안 생성]
     │
     ├─→ [유효 노출량 시뮬레이션 (현재 vs 개선안)]
     │
     └─→ [스토어명 연동 보너스 키워드 분석]
            │
            ▼
     [최종 리포트 출력]
      - 현재 상품명 점수/등급
      - 개선 상품명 제안
      - 유효 노출 키워드 변화
      - 예상 검색량 증감
```
